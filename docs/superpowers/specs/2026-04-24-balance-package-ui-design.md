# 余额包 UI 改版设计（用户购买页）

## 背景

当前余额包区域已经能完成基础购买，但整体更像配置列表，缺少“商品卡片”的购买感；同时线路限制信息不够前置，用户容易在点击后才知道是否可用。

本次改版目标是把余额包区域改成更接近商城卡片的购买体验，同时保留线路约束的清晰表达，降低误购概率。

## 目标

1. 让余额包更像“可购买商品”，提升第一眼可理解性和购买意愿。
2. 把 `Codex / 通用` 线路信息前置到卡片核心视觉层级。
3. 把“支持哪些分组 / 是否可混用”做成强提示，而不是埋在描述里。
4. 将“选择商品”和“确认支付”拆成两步，降低单屏信息压力。
5. 保持现有下单链路、支付方式、业务规则不变，仅优化前端展示与交互。

## 不在本次范围内

- 不调整余额包后端数据结构。
- 不修改购买、支付、回调、到账的业务逻辑。
- 不引入新的支付方式。
- 不重做整页充值/订阅架构，仅重做余额包区域及其确认态。

## 用户问题总结

当前用户在余额包区域会遇到这些体验问题：

- 卡片缺少商品感，价格、到账额度、线路之间的主次不清。
- 支持范围不够显眼，容易买错线路。
- 列表和确认信息混在一起，认知负担偏高。
- 已有线路余额时，限制规则提示不够靠前。

## 选定方向

采用“**商城卡片风**”作为主方向，具体落点为：

- 视觉骨架采用“**A：转化优先**”方案。
- 同时吸收“**B：信息平衡**”里对支持范围的清晰表达。
- 优先强调“**支持范围**”，因为该业务最核心的风险是误购错误线路。

## 信息层级

余额包卡片的视觉优先级按以下顺序排列：

1. `Codex / 通用` 线路标签
2. 余额包名称
3. 到账额度（`$xx`）
4. 售价（`¥xx`）
5. 支持范围提示
6. 描述文案

说明：到账额度比人民币价格更接近商品本体价值，因此在列表态中优先级高于价格；但价格仍需足够清晰，不能弱化到用户找不到。

## 页面结构设计

### 1. 顶部区域

保留现有账户信息，但在余额包区域前增加更明确的线路状态提示。

建议结构：

- 当前账号
- 当前余额
- 当前余额线路（若存在）
- 线路限制提示

示例文案：

- 当前余额线路：Codex
- 仅可继续购买 Codex 余额包

如果用户当前没有已激活线路，则不显示限制性语气，只显示常规说明。

### 2. 余额包列表态

采用一组商品卡片展示余额包。每张卡片包含：

- 顶部左侧：线路标签（Codex / 通用）
- 顶部右侧：到账额度 `$xx`
- 中部：余额包名称
- 中部副文案：一句简短说明（来自 description，必要时裁切）
- 下部：售价 `¥xx`
- 支持范围提示块
- 底部主按钮：`立即购买`

### 3. 支持范围提示块

支持范围不再只放在描述文案里，而是做成独立信息块。

示例内容：

- 仅支持 Codex 分组
- 不可与其他线路余额混用

或：

- 仅支持通用分组
- 可用于通用线路对应分组

此信息块要明显，但不要做成报错样式；应更像购买须知。

### 4. 余额包确认态

当用户点击卡片或按钮后，不跳转页面，而是在当前页切换到确认态。

确认态展示：

- 已选余额包名称
- 线路标签
- 到账额度
- 售价
- 支持范围说明
- 购买限制说明
- 支付方式选择
- 主按钮：`立即支付 ¥xxx`
- 次按钮：返回列表

列表态负责“选哪个”，确认态负责“确认规则并付款”。

## 视觉风格设计

### 1. 整体气质

采用“**轻电商 + 控制台可信感**”的风格：

- 不是活动海报风
- 不使用夸张营销视觉
- 保持整洁、可信、专业
- 但明确体现商品卡片感

### 2. 标签设计

- `Codex` 使用蓝色系标签
- `通用` 使用紫色或青色标签
- 标签使用圆角 badge 样式
- 放在卡片左上角高可见区域

其作用是快速识别线路，而不是纯装饰。

### 3. 金额设计

- 到账额度 `$xx` 作为第一视觉重点
- 售价 `¥xx` 作为第二重点
- 在单张卡片中，两者必须同时明显，但到账额度更醒目

### 4. 卡片状态

#### 默认态

- 白底 / 深色模式对应深底
- 中等圆角
- 干净边框
- 轻阴影

#### Hover 态

- 轻微上浮
- 阴影增强
- 边框颜色略强化

#### 选中态

- 边框高亮
- 背景轻度染色
- 主按钮更强调

#### 不可购买态

- 整卡轻灰化
- 禁用鼠标态
- 按钮显示不可购买
- 明确显示原因

## 交互流程设计

### 阶段 1：选择商品

用户在列表态浏览并选择余额包。

页面重点是帮助用户快速比较：

- 哪条线路
- 到账多少
- 售价多少
- 能不能用于自己的分组

### 阶段 2：确认购买

用户选中后进入确认态，再选择支付方式并发起支付。

这样做的原因：

- 列表页不需要塞入过多支付信息
- 用户先完成“选商品”决策
- 支付确认阶段再补充完整规则和支付操作

## 约束与禁用态设计

### 1. 当前账户已有线路余额时

在列表上方展示状态提示，并在不兼容的卡片上直接体现为不可购买。

示例：

- 当前余额线路：通用
- 当前不可购买 Codex 余额包

### 2. 不兼容卡片

如果某余额包与当前用户线路冲突：

- 卡片不可点击
- 按钮显示：`当前不可购买`
- 卡片内明确展示原因，如：
  - 当前账户已有通用线路余额
  - 请先消耗完现有余额后再切换线路

原则：尽量在列表层解决理解问题，不把错误留到提交后。

## 组件与实现建议

在现有 `frontend/src/views/user/PaymentView.vue` 基础上进行局部重构，优先考虑拆分以下子块：

- 余额包列表卡片组件（可选）
- 余额包顶部状态提示块
- 余额包确认摘要块

如不拆单文件组件，也应至少在 `PaymentView.vue` 内通过计算属性和小型渲染区块降低模板复杂度。

## 数据与逻辑约束

以下业务逻辑保持不变：

- 余额包仍来自现有 `checkout.balance_packages`
- 支付方式仍使用当前 `enabledMethods / methodOptions`
- 购买调用仍走现有 `confirmBalancePackage / createOrder`
- 余额线路冲突规则仍由后端兜底
- 前端只新增更明显的展示与禁用态判断

## 错误处理

前端应把错误尽量前置，但仍保留后端错误兜底：

- 若后端返回 `PACKAGE_SCOPE_CONFLICT`，仍弹出现有错误提示
- 若用户在旧状态下进入页面导致前端判断失真，以后端结果为准
- 不因本次 UI 改版改变错误码与支付失败处理流程

## 测试建议

至少补充或更新以下前端测试：

1. 余额包列表正确展示线路标签、到账额度、价格。
2. 存在余额包时默认进入余额包 tab。
3. 余额包卡片选中后进入确认态。
4. 确认态展示已选余额包关键信息。
5. 不兼容余额包显示为禁用态（如当前已有其他线路余额）。
6. “充值” tab 继续保持不可点击。

## 完成标准

本次设计落地后，应达到：

- 用户第一次进入余额包页时，能迅速看懂每张卡卖的是什么。
- 用户在购买前就能知道它支持哪类分组。
- 页面更像商品购买页，而不是配置列表。
- 不改变既有支付主流程。

## 推荐实施顺序

1. 重构余额包列表卡片 UI
2. 增加顶部线路状态提示
3. 重构确认态摘要区域
4. 增加不可购买卡片态
5. 更新 Vitest 用例并做构建验证
