# Upstream v0.1.126 合并保护设计

## 背景
本地 `main` 已包含一组 fork 自有功能，重点包括：余额套餐、`package_scope`、强制切换余额模式、支付页/支付结果页改造、首页套餐展示，以及已落地的返利相关行为。上游 `upstream/main` 已推进到 `f5bd25bea`（v0.1.126 对应区间），直接合并时在支付链路、前端类型、Ent 生成代码上出现 7 个冲突文件。

本次工作的最高原则不是“尽快追上上游”，而是“在吸收上游更新时，不回退任何现有功能”。

## 目标
在独立合并分支中完成一次可回放、可验证的上游同步：
- 保留本地现有功能和现有用户体验
- 吸收上游最新稳定更新
- 将冲突收敛为可验证的几个批次处理
- 在没有完整证据前，不改动当前工作中的 `main`

## 非目标
- 不借这次合并做无关重构
- 不顺手重写支付架构、返利架构或 Ent 结构
- 不为追求“更像上游”而牺牲现有行为
- 不在当前主工作目录直接操作半合并状态

## 方案选择
采用“隔离分支直接合并上游，再逐个按保现有功能原则解冲突”的方案。

放弃“只 cherry-pick 部分提交”与“以上游为底重贴本地补丁”的原因：
- 前者容易漏掉依赖提交，后续继续同步上游会更难
- 后者对现有功能回退风险最高，不符合本次保护目标

## 保护边界
以下能力视为必须保留，解冲突时默认优先保住：
1. 余额套餐购买能力
2. `package_scope` / 套餐模式区分
3. 强制切换余额模式流程
4. 支付页与支付结果页的现有展示与交互
5. 首页套餐展示相关改动
6. 已存在的返利、返现、返利明细相关行为
7. 与上述能力直接相关的后端订单、类型、数据库字段

## 冲突分组与处理顺序
### 第一组：前端显式行为保护
文件：
- `frontend/src/views/user/PaymentView.vue`
- `frontend/src/views/user/PaymentResultView.vue`
- `frontend/src/types/index.ts`

处理原则：
- 以本地现有交互为主，先保住余额套餐、模式切换、支付结果展示
- 再按需吸收上游新增字段、空值保护、兼容性修复
- 对 `types/index.ts` 采用“功能并集”方式，不能因为接受上游版本而丢掉本地类型字段

### 第二组：后端订单与支付语义保护
文件：
- `backend/internal/service/payment_order.go`

处理原则：
- 以本地“余额套餐 + scope 切换”语义为主线
- 将上游在支付主题、金额处理、兼容行为上的修复逐条并回
- 不允许出现：本地订单类型失效、字段不再写入、强制切换逻辑被绕过、支付结果口径变化

### 第三组：Ent 生成代码对齐
文件：
- `backend/ent/migrate/schema.go`
- `backend/ent/mutation.go`
- `backend/ent/runtime/runtime.go`

处理原则：
- 这些文件不单独“手工设计行为”，只服务于 schema 真相
- 以当前仓库真实 schema 为准，必要时重新生成或按 schema 对齐
- 重点防止索引位移、字段序号变化、默认值/校验器遗漏导致的隐性运行时错误

## 执行流程
1. 在隔离 worktree 分支中重新进入合并态
2. 先解决第一组前端冲突，并运行前端定向测试
3. 再解决第二组后端订单冲突，并补充/运行支付相关定向测试
4. 最后解决第三组 Ent 生成代码，使其与 schema 真相一致
5. 运行能在当前环境完成的完整校验；若外部工具链受限，至少保留定向验证和阻塞证据
6. 仅在验证通过后，才把该分支作为候选合并结果

## 验证策略
### 必做验证
- 支付页关键测试：选择余额套餐、切换模式、下单参数构造
- 支付结果页关键测试：手续费为空、余额套餐金额前缀、展示容错
- 后端订单关键测试：`balance_package`、`balance_package_id`、`package_scope_snapshot`、`force_switch_scope` 路径
- 合并后再次检查是否还存在冲突标记或遗漏字段

### 环境受限说明
当前环境里后端完整测试受 Go 1.26.2 工具链下载超时阻塞。因此验证分两层：
- 第一层：本地可立即完成的前端与静态/差异验证
- 第二层：一旦工具链可用，补跑完整 `make test` 或至少后端支付相关测试

### 完成标准
只有同时满足以下条件，才视为“这次合并可以继续推进”：
- 7 个冲突文件全部解完
- 本地必须保留的现有功能点都能被对应测试或代码证据覆盖
- 前端关键测试通过
- 后端关键路径至少有定向测试或可读证据证明未回退
- worktree 中不存在未解释的额外改动

## 风险与回退策略
### 主要风险
- 接受上游生成代码后，悄悄覆盖掉本地 schema 字段
- 只让代码“编译过”，但支付语义已变
- 前端类型被上游简化，导致本地页面逻辑静默退化

### 应对方式
- 冲突按“前端行为 → 后端语义 → 生成代码”顺序处理，避免先被生成文件带偏
- 每一组处理完都做定向验证，不等到最后一起看
- 保持当前 `main` 不动；失败时直接丢弃隔离分支即可

## 交付物
本次设计输出的交付物包括：
- 一条隔离合并分支
- 一组带验证证据的冲突解决提交
- 一份明确说明“保住了哪些现有功能、吸收了哪些上游修复”的总结
