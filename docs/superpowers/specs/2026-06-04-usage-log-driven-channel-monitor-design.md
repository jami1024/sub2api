# 使用日志驱动的渠道监控设计

## 背景

当前渠道监控会通过定时任务或手动“运行一次”主动请求上游 endpoint，并把请求结果写入监控历史。用户希望渠道监控不再额外发起探测请求，而是直接根据真实用户请求产生的使用日志展示状态。

本次需求范围限定为 OpenAI 渠道监控场景。用户确认：

- 不新增渠道绑定字段。
- 只会增加一个 OpenAI 渠道。
- 自动定时监控不需要主动请求。
- 手动运行也不需要主动请求。
- 没有真实使用日志时，不应显示失败；没有请求就没有状态。

## 目标

1. 渠道监控自动运行不再请求上游。
2. 渠道监控手动运行不再请求上游。
3. 监控状态优先来自最近真实使用日志。
4. 没有匹配使用日志时，状态保持未知/空，不误报失败。
5. 尽量保留现有 API 响应结构和前端显示逻辑，减少兼容风险。

## 非目标

- 不新增 `channel_id` 或其他绑定字段。
- 不做多渠道、多 endpoint 精确匹配。
- 不删除现有 monitor 配置字段，例如 `endpoint`、`api_key`、`interval_seconds`。
- 不改真实网关请求、计费、使用日志写入逻辑。
- 不用缺失日志推断渠道不可用。

## 状态来源

渠道监控状态改为读取 `usage_logs`。

匹配维度：

- 仅处理 `provider = openai` 的监控。
- 按监控模型匹配使用日志中的模型字段。
- 可同时考虑 `requested_model`、`model`、`upstream_model`，避免渠道映射后模型名不同导致找不到日志。
- 优先读取成功使用日志。当前仓库已有统计口径使用 `actual_cost > 0` 作为成功请求代理，本功能沿用该口径。
- OpenAI endpoint 可作为辅助过滤：`inbound_endpoint` / `upstream_endpoint` 中包含 OpenAI chat、responses 或 compact 相关路径时视为 OpenAI 使用记录。

## 状态判定

对每个监控模型读取最近一条匹配的成功使用日志：

- 找到成功日志：
  - 默认状态为 `operational`。
  - 如果日志有 `duration_ms`，并且超过现有慢响应阈值 `monitorDegradedThreshold`，状态为 `degraded`。
  - `latency_ms` 使用该日志的 `duration_ms`。
  - `checked_at` 使用日志的 `created_at`。
  - message 留空，或使用简短说明“derived from usage log”。
- 没有匹配日志：
  - 状态为空/未知。
  - 不写 `failed` 或 `error`。
  - 不生成误导性的历史失败记录。

## 数据流

### 列表状态

1. 后台打开渠道监控列表。
2. 后端读取 monitor 配置。
3. 对 OpenAI monitor 的主模型和附加模型，从 `usage_logs` 查询最近成功日志。
4. 返回 `primary_status`、`primary_latency_ms`、`extra_models_status`。
5. 没有日志时保持空状态。

### 手动运行

1. 管理员点击“运行一次”。
2. 后端不请求上游。
3. 后端读取最近使用日志并返回合成的 `results`。
4. 没有日志的模型返回空状态或未知状态，不返回失败。

### 自动定时

1. runner 不再发起上游 HTTP 探测。
2. 可选择不再调度检测任务，避免周期性写入重复历史。
3. 状态展示直接读使用日志，保证后台刷新不会造成额外请求。

## 历史与可用率

为了避免“同一条真实使用日志被定时器重复写成多条监控历史”，本次不再从自动定时任务生成探测历史。

历史页面可以保留原有历史数据；新的日志驱动状态不强制写入 `channel_monitor_histories`。

可用率如果没有真实监控历史，可保持现有历史聚合结果；没有历史时返回 0 或空状态。后续如需要更准确的日志可用率，可以单独做二期：按使用日志窗口计算成功率。

## 错误处理

- 查询使用日志失败时，不请求上游兜底。
- 列表状态保持空，避免误报失败。
- 记录后端日志，便于排查查询失败。
- 非 OpenAI provider 暂不做日志推断，保持空状态或原有历史状态。

## 测试计划

后端单元测试：

1. OpenAI monitor 有最近成功日志时，主模型状态为 `operational`。
2. 最近成功日志耗时超过阈值时，状态为 `degraded`。
3. 没有匹配日志时，不返回 `failed` / `error`。
4. 手动运行不调用 `runCheckForModel` / 不发 HTTP 请求，只返回日志推断结果。
5. 自动 runner 不再因定时任务发起上游请求。
6. 模型匹配覆盖 `requested_model`、`model`、`upstream_model`。

回归验证：

- `go test ./internal/service -count=1`
- 如涉及 repository SQL，补充对应 repository 单元测试或集成测试。

## 兼容性

- 不新增数据库字段。
- 不删除已有字段。
- 不改变 monitor 创建/编辑表单的基础结构。
- 不影响真实网关请求与计费逻辑。
- 旧的监控历史保留，只是不再通过主动探测产生新历史。
