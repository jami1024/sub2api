# 管理员供应商状态页设计

## 背景

当前系统已经有面向用户的渠道监控页，也有运维监控页，但缺少一个只给管理员看的“按供应商看整体状态”的页面。你希望这个页面直接基于真实使用记录统计，不再主动探测上游。

该页面只供管理员查看，普通用户不应看到入口，也不应拥有访问路径。

## 目标

1. 新增一个管理员专用页面，用于查看各供应商整体情况。
2. 数据完全来自真实使用记录，不主动请求上游。
3. 支持按时间范围查看供应商的可用率、请求量、错误率和延迟分布。
4. 页面结构尽量接近截图中的运维风格，但只保留第一版必要信息。
5. 普通用户不可见、不可访问。

## 非目标

- 不做用户侧页面改造。
- 不做主动探测或健康检查任务。
- 不新增供应商绑定配置。
- 不做账号级、endpoint 级、模型级的多层下钻。
- 不做复杂告警规则或自动修复。

## 页面范围

新增页面建议放在管理员菜单下，路由示例为：

- `/admin/provider-status`

页面仅显示供应商名称聚合结果，不额外要求用户手动分组。

## 数据来源

页面只读使用记录相关数据：

- 成功请求：`usage_logs`
- 失败请求：`ops_error_logs`
- 供应商归属：优先从使用记录关联到的账号/分组平台信息获取；若无法稳定归属，则回退到可识别的供应商名称或 `unknown`

供应商名称的优先级建议为：

1. 账号/分组平台名
2. 记录中的上游供应商标识
3. `unknown`

## 指标定义

每个供应商至少展示：

- 总请求数
- 成功请求数
- 失败请求数
- 可用率
- 错误率
- P50 / P95 / P99 延迟
- 最近请求时间
- 供应商时间线

建议可用率口径：

- `可用率 = 成功请求 / (成功请求 + 非业务限制失败请求)`

业务限制类失败不计入供应商不可用，例如：

- 余额不足
- 配额超限
- 参数错误
- 由用户输入或配置导致的明确拒绝

应计入供应商异常的情况包括：

- 上游 5xx
- 超时
- 连接失败
- 上游限流
- 其他明显的上游可用性问题

## 页面布局

第一版采用三块结构：

1. 顶部筛选条
   - 时间范围：15 分钟、1 小时、6 小时、24 小时、7 天
   - 刷新按钮
2. 供应商总览表/卡片
   - 每行一个供应商
   - 显示请求数、可用率、错误率、P50/P95/P99、最近请求时间
   - 带简短时间线标记
3. 趋势区域
   - 延迟分布或延迟趋势图
   - 重点展示 P50 / P95 / P99

## 交互

- 点击时间范围后刷新整页数据。
- 点击刷新按钮重新拉取数据。
- 第一版不做明细钻取。
- 供应商无数据时显示“暂无数据”，不要误显示为失败。

## 后端设计

建议新增一组管理员只读接口，服务端聚合后直接返回页面所需结果。

建议形式：

- `GET /api/v1/admin/provider-status/summary`
- `GET /api/v1/admin/provider-status/timeline`

如果实现上更简单，也可以只保留一个聚合接口，返回：

- 供应商列表
- 当前窗口汇总
- 趋势点

后端计算优先复用现有使用记录统计逻辑，不重复造一套探测链路。

## 前端设计

建议新增一个管理员页面组件，风格与现有运维页一致：

- 使用现有布局组件
- 复用时间范围选择器、表格样式、图表样式
- 通过管理员侧边栏进入
- 对普通用户隐藏入口

## 错误处理

- 统计查询失败时显示空状态或错误提示，不回退主动探测。
- 单个供应商统计失败不应影响整页渲染。
- 缺少归属信息时统一归到 `unknown`，避免丢数据。

## 测试计划

后端：

1. 按时间范围返回供应商汇总数据。
2. 业务限制错误不计入可用率分母。
3. 上游错误计入不可用。
4. 无数据时返回空列表而不是失败。
5. 管理员接口只对管理员路由开放。

前端：

1. 页面可正常加载并展示供应商列表。
2. 时间范围切换会重新请求数据。
3. 无数据时显示空状态。
4. 用户侧路由和菜单不暴露该页面。

## 兼容性

- 不改现有用户侧渠道监控页。
- 不影响现有使用记录写入。
- 不影响现有运维仪表盘和告警页面。
- 新页面只读，不新增业务写入路径。
